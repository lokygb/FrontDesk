Enhancements

Stability
  • The system must be absolutely secure and stable. It is not an option to go down for any extended period of time, or be possible to compromise te integrity of the system.
  • The system must improve its robustness. There should be no point in which the integrity of the system is put into question as a result of invalid input percolating to the database level
Student Interface
  • Modes of Submission
    • Archive Based - Student creates a ZIP, JAR, TAR, etc. file and sends this code to the system for processing.
    • CVS Based - Student maintains CVS module on the handin server. Upon completion of the assignment, the student can choose to submit final work from handin CVS module instead of creating an archive.
  • Modes of Evaluation
    • Students will then be told they will receive an email indicating pre-test and compile correctness. This will allow the server to amortize the cost of doing these pre-tests instead of trying to run them all at once to give the student instant feedback
    • TAs can also enable the instant feedback on the Web page if they want.
  • More detailed information about handin difficulties (such as compile problems) available to students. A student should be able to access the Directory Viewer interface to attempt to compile code and to get more information about why their submission failed.

Testing Interface
  • Server Side Testing
    • Testing Centers run the run the server side testing software
    • Potentially many testing centers available to the handin server. Main testing center will delegate certain tests to any free testing center.
    • Data can be either in a shared disk space, or if this is impossibly, can be transmitted to the remote centers.
    • Increased amount of status information on jobs. More robust ability to kill jobs and to even get a snapshot of what exactly is happenning
    • JUnit
    • Tests
      • Sandboxed test area on the server for greater security
      • All tests will conform to testing interface. Scores for tests will *not* be determined by the system, instead the test program submitted will generate the score
      • Test progroms will follow a well defined specification, most likely communicating with testing server through XML based communication
      • The system will provide the means for sandboxing, running against multiple students, etc, while the test program is entirely responsible for assigning the scores
  • Testabililty
    • Input/Output Guidelines for student code
    • Logging guidelines (vs. System.out.println).
    • 211 Standard Library (user I/O, logging, file I/O, GUI (?))
  • Client Side Testing
    • Leverage the power of the testing center to provide students with a means to self-grade their own submissions. Provides more feedback and also provide
    • Client side testig architecture can provide a semi-secure way of distributing testing across the students. Spot check testing can be used to make sure no one is abusing the system.
  • Abstractions between test programming language and underlying testing operating system.
    • Such abstraction will allow high level components to do testing related activities without restricting the system to a particular programming language or operating system.
    • Feasability assesment for a specific other language (to be selected (C++?))

Grading Interface
  • Directory Viewer
    • WebDAV
    • Addresses problem of only server admins being able to modify handin internal directory structure
    • "Windows Explorer" type access to the students submission directory
    • Ability to edit code, change directory structure, attempt to build, etc.
  • More result viewing options. Summary views, entire section views, entire section prinout ability
  • Modify objective test scores, add in a subjective test score and extra credit points that are included in the students results for that assignment.
  • Diff the user and correct output to zero in on differences in complex output (CVS Diff)
  • Code markup can be annotated with "handin markup tags"
    • Specifically comments from TA, so that it is highlited or decorated in some way so that when a student views their code, they can easily see TA comments. These tags would be applied int he Directory Viewer module.
    • Comments/Tags can be summarized for students and TAs to view quickly

Admin Interface
  • More control over your section. Geared toward section management (particulary your own section). Send email, view entire section grades, view grading/submission information (summary), etc.
  • TAs should be presented with a section oriented view upon logging into the system
  • Backups and undo capability (see Plone)
